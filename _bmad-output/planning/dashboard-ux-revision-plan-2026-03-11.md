---

## 1. 修订目标

这份文档不是重做 UX，而是把现有 dashboard 规格从“方向正确”收束到“可以稳定落地”。

修订目标：

1. 明确 dashboard 首页边界，避免信息过载
2. 固定 KPI、筛选和刷新规则，消除实现歧义
3. 补强表格与异常场景设计，让用户能从总览走到确认
4. 为后续视觉设计和前端实现提供更具体的页面约束

---

## 2. 页面定位

### 2.1 首页角色

`/dashboard` 首页应被定义为：

**监控总览页，而不是全功能分析工作台。**

它的核心任务只有 3 个：

1. 让运维/分析人员在 5 秒内判断系统是否健康
2. 让用户快速发现异常发生在哪个时间段、哪个维度
3. 让用户从异常概览进入记录级验证

### 2.2 不应该在首页承载的内容

以下内容不应作为首页主模块：

- 复杂调试视图
- 过深的请求详情分析
- 大量可配置的个性化布局
- 低频管理功能

这些能力如果需要，应该进入二级页面或抽屉。

## 3. 修订后的信息架构

建议将首页固定为 5 个层级，自上而下如下：

### 3.1 全局控制栏

包含：

- 页面标题：`Gateway Overview`
- 最近更新时间
- 时间范围选择器
- 自动刷新开关
- 导出按钮

规则：

- 默认时间范围：`最近 24 小时`
- 自动刷新默认开启，仅对预设时间范围生效
- 当用户选择自定义时间范围时，自动刷新自动关闭

### 3.2 健康状态摘要区

只保留 4 张核心 KPI 卡片：

1. 总请求数 `Total Requests`
2. 错误率 `Error Rate`
3. P95 延迟 `P95 Latency`
4. Fallback 次数 `Fallbacks`

说明：

- 不建议首页首屏直接放成本卡片
- 成本更适合进入趋势区或排行区的次级信息
- 如果后端暂时没有 P95，仅可临时使用 `Avg Latency`，但需要在 UI 中明确标注口径

### 3.3 趋势观察区

固定两张图，不再扩张：

1. 请求量趋势
2. 错误率 / 延迟趋势

规则：

- 两张图必须共享同一时间轴
- Hover 时联动 tooltip
- 默认展示最近 24 小时
- 时间粒度随范围自动切换：
  - 24 小时：按小时
  - 7 天：按天
  - 30 天：按天或周

### 3.4 分析筛选区

筛选项保留 4 个主维度：

- Team
- Router
- Channel
- Model

规则：

- 筛选默认即时生效
- 所有已生效筛选必须以 chips 形式显式展示
- 提供 `Reset all` 清空入口
- 筛选状态应写入 URL query，支持分享和刷新恢复

### 3.5 Usage 验证区

这是首页最重要的下层模块，用来验证异常，不是装饰性表格。

默认字段建议固定为：

- Time
- Team
- Router
- Channel
- Model
- Total Tokens
- Latency
- Status

规则：

- 默认按时间倒序
- `Status` 至少区分 `Success / Error / Fallback`
- 行点击打开右侧详情抽屉
- 抽屉展示请求元信息，不在首页内直接展开复杂内容

## 4. KPI 口径修订

现有规格中，卡片指标与后端 API 定义存在轻微错位。为避免实现阶段反复讨论，建议固定如下：

| UI 指标 | 建议数据源 | 说明 |
|------|------|------|
| Total Requests | `/api/metrics.total_requests` | 首页核心总量指标 |
| Error Rate | `total_errors / total_requests` | 需前后端统一保留两位小数 |
| P95 Latency | 新增或聚合查询 | 若暂缺，先显示 Avg Latency 并标注 |
| Fallbacks | `/api/metrics.total_fallbacks` | 反映路由稳定性 |

不建议首页首屏直接展示：

- Success Rate
- Cost Today
- Token Total

这些都可以在趋势图、详情抽屉或二级分析区呈现。

## 5. 关键交互规则

### 5.1 时间范围

推荐预设：

- `24H`
- `7D`
- `30D`
- `Custom`

行为规则：

- 切换时间范围后，KPI、图表、表格同步刷新
- 自定义范围需要显示起止日期
- 当所选范围超过 30 天，可降低图表精度，优先保证可读性

### 5.2 自动刷新

规则：

- 仅在 `24H / 7D` 预设下可开启
- 默认间隔 `30s`
- 用户手动应用筛选时，不中断当前视角
- 刷新过程中保留当前滚动位置和筛选状态

### 5.3 导出

导出不是主路径，但必须明确：

- 导出对象为当前筛选结果
- 导出格式先支持 CSV
- 点击后给出 loading 状态
- 失败时用 toast + 内联错误提示说明原因

## 6. 异常场景补强

首页必须对以下 4 类异常给出明确设计，不可只写“错误提示”：

### 6.1 无数据

场景：

- 新部署实例
- 当前筛选无命中

区分规则：

- 新实例无数据：引导说明“尚未产生请求”
- 筛选无结果：提示“当前筛选无匹配结果”，并提供 `Reset filters`

### 6.2 数据拉取失败

场景：

- API 不可用
- 网关短时异常

设计要求：

- KPI 区与图表区显示降级态
- 说明“无法获取最新数据”
- 提供 `Retry`
- 保留上次成功数据的时间戳

### 6.3 指标异常飙升

场景：

- 错误率激增
- Fallback 连续升高

设计要求：

- 顶部出现警示条
- 异常卡片进入高优先级视觉状态
- 图表上对异常区间进行高亮

### 6.4 权限不足

场景：

- 某些 Team 或 Router 不可见

设计要求：

- 明确显示“部分数据受权限限制”
- 不要把权限问题混淆成空数据

## 7. 视觉方向修订

现有规格的视觉方向偏稳妥，但略通用。建议保留冷静、可信赖的监控风格，同时减少“模板式 SaaS 面板”的感觉。

### 7.1 视觉原则

- 首屏优先读数，不优先装饰
- 用色服务状态，不服务炫技
- 异常视觉优先级必须高于品牌色

### 7.2 颜色优先级

建议建立明确优先级：

1. 错误 / 告警色
2. 数据主趋势色
3. 品牌强调色
4. 中性层级色

这可以避免界面里“到处都在强调”的混乱感。

### 7.3 字体建议

当前文档使用 `Inter` 没问题，但如果希望 dashboard 更有识别度，可以只在页面标题或大数字上增加一层更有张力的字形处理，而不是全局换字体。

## 9. 落地优先级

### P1 必须先定

- 首页边界
- 4 个 KPI 定义
- 时间筛选规则
- 表格字段和排序
- 异常状态样式

### P2 随后补强

- 趋势图多时间粒度
- 行详情抽屉
- 导出反馈
- URL 状态同步

### P3 可后续迭代

- 成本分析增强
- 排行榜模块
- 保存常用筛选
- 个性化 dashboard

---

## 10. 下一步建议

如果继续推进，最合理的动作不是再写概念，而是直接进入下面两项之一：

1. 产出 `dashboard 页面结构说明`，把每个模块拆成组件级规格
2. 产出 `dashboard 低保真线框说明`，直接指导视觉和前端实现

我建议先做第 1 项，因为它能把设计和实现对齐得最快。
